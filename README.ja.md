# PV AE マスタージェネレーター＆シグナルモニター (PV AE Master Generator & Signal Monitor)

AIを活用した最新のファーマコビジランス（PV）Webアプリケーションです。添付文書からの有害事象（AE）データ抽出、その添付文書に対する四半期ごとの発現率シグナル検出、そして——PV-Linkプロジェクトとの統合後に加わった——公表文献（PubMed）を監視して新たな安全性シグナルをCIOMS/E2Bドラフトまで追跡する、というPVシグナル検出ワークフロー全体をカバーします。

## 🌟 主な機能

上部ナビゲーションには8つの項目があり、**NEWと付いている項目**はPV-Linkから統合された文献モニタリングのワークフローです。

1. **📋 ダッシュボード（总覽）**
   - 製品ごとの四半期モニタリング進捗をひと目で確認でき、今四半期まだ対応が済んでいない製品へはショートカットで直接アクセスできます。
   - **NEW** — 「文献の未レビュー」件数を示すリマインダーバッジを表示し、クリックすると文献レビューへ直接移動します。

2. **🤖 AIによるAE抽出（ジェネレーターモード）**
   - PDF、画像、テキスト形式の添付文書をアップロードできます — 単一ファイルでも、**複数ファイルの一括処理キュー**（ファイルごとの再試行に対応した逐次抽出）でも対応。
   - 複数のAIプロバイダーに対応しており、単一の統合LLMクライアント（`services/llm.ts`）を通じて解決されます: **プラットフォームデフォルト**（キー不要）、**自前のGemini**（構造化出力）、または任意の**自前のOpenAI互換エンドポイント**（OpenRouter、Ollama、LM Studioなど）。
   - データベースへ保存する前に、AIが抽出したデータを確認・修正できる編集可能なデータグリッドを提供します（一括処理の結果も保存前に人によるレビューが必要です）。

3. **🔎 文献検索（NEW）**
   - 対象の有効成分・AEキーワード・除外語から組み立てられる、正確かつ再現可能な**PubMed E-utilities**検索（esearch + efetch）。日付フィルタは出版日ではなく、**PubMedへの収載日（edat: entry date）**——文献がPubMedに追加された日——を基準にしており、モニタリング期間中に新たに現れた文献を確実に捕捉します。
   - ヒットしたすべての文献に対し、AIによる関連性スコア（0〜100）と繁体字中国語の要約・結論が生成されます。これらはバッチ化され、統合LLMクライアントを通じて並行実行されます。
   - 検索条件（件数上限、日付範囲、AEキーワード、除外語）はセッションをまたいで保持され、毎回入力し直す必要があるのは対象成分のみです。

4. **✅ 文献レビュー（NEW）**
   - AIによるスコア付け・要約が済み、人による判断待ちの文献キュー。最低関連性スコアで絞り込み可能です。
   - 選択すると、構造化されたPVデータ（製品名、成分、有害事象の原文表記、MedDRA基本語(PT)候補、重篤性、因果関係、転帰など）が各レコードにつき一度自動抽出されます。
   - **文献ライブラリへ取り込む**、または**却下する**——どちらの操作も監査証跡に記録されます。レコードごとに**CIOMS-I / E2B(R3)ドラフト**を生成・閲覧・コピー・ダウンロードでき（抽出データからの決定論的なフィールドマッピングであり、あくまでPV担当者によるレビューが必要なドラフトの骨子です）。

5. **📊 シグナル検出とモニタリング（モニターモード）**
   - 販売数量と1日投与量から四半期ごとの曝露量を計算します。四半期ごとのAE症例数を入力できます（行ごとに**重篤**フラグ付き）。
   - 実際の発現率を、正確法（Garwood法）による**ポアソン分布の95%信頼区間**とともに添付文書の閾値と比較し、**予期せぬ事象 (Unexpected)**、**アラート (Alert)**、**警告 (Warning)** のシグナルをフラグ付けします。
   - **設定可能なシグナル判定ルール**（最小症例数、アラート倍率、許容バッファ）——各レポートには使用したルールが記録され、ノイズとして抑制された行には注記が付きます。あいまい一致（＋AIによる同義語正規化パス）により、未一致のAE入力に対して最も近いマスター用語を提案します。
   - CSVへのエクスポート、または判定ルールの記載と署名欄を含む印刷向けの四半期レポート（PDFとして保存可能）を出力できます。
   - **NEW** — 四半期レポートの下に、独立した2つめのパネル「文献シグナルモニタリング」を追加。**有効成分 × MedDRA基本語(PT)**で文献ライブラリを集計し、キーワードで絞り込めます。これは定性的な文献件数の集計であり、上記のポアソン信頼区間分析とは異なる方法論です（両者を並べて表示することでこの違いを明示しています）——統計的有意性検定ではありません。

6. **📚 文献ライブラリ（NEW）**
   - 文献ライブラリ（レビューモードから取り込まれたレコード）の閲覧、キーワード／日付での絞り込み、CSVエクスポートができます。
   - 構造化PVデータが未抽出のレコードに対して**一括構造化抽出**を実行でき、進捗インジケーターが表示されます。
   - ライブラリからレコードを削除できます（監査ログに記録）。クリックしてレビューモードで該当レコードを再度開くこともできます。

7. **🗄️ マスターデータ管理（ライブラリモード）**
   - 保存されたすべての製品AEマスターを一元管理するデータベースです。
   - 過去の四半期モニタリングレポートとシグナル検出のバッチを閲覧できます。
   - **四半期をまたいだトレンド分析**: AEごとの発現率の推移を折れ線グラフで表示し、2四半期以上連続で上昇している用語には自動でフラグが付きます。
   - **マスターのバージョン履歴**: 更新のたびに旧マスター行がアーカイブされ、追加／削除／閾値変更を確認できる差分ビューアが利用できます。
   - **JSON形式の完全バックアップのエクスポート／インポート**——バックアップのスキーマには文献ライブラリと未レビューキューも含まれるため、1つのファイルからすべてを復元できます。

8. **🛡️ 監査証跡（監査モード）**
   - GVPに準拠したシステムログで、**SHA-256ハッシュチェーン**を採用しています。各エントリは直前のエントリと連結されており、UI側でチェーンの整合性を検証するため、改ざんや削除を検知できます。
   - `CREATE`、`UPDATE`、`DELETE`、`EXPORT`、`IMPORT`、`ANALYSIS`のすべての操作を、文献ワークフローを含む全モジュールにわたって記録する読み取り専用の監査証跡です。

**⚙️ 設定**（右上の歯車アイコン）: AIプロバイダー／モデル／キーの設定と、シグナル判定ルールの調整を行えます——詳細は後述の**AIプロバイダーに関する注意**を参照してください。

## 🚀 はじめに（ローカル開発）

### 前提条件
- Node.js (v18以上を推奨)
- アプリを試すのにAI APIキーは不要です — プラットフォームデフォルトのプロキシはキーなしで動作します。特定のプロバイダー／モデル（Google Gemini、OpenRouter、ローカルのOllamaなど）を使いたい場合のみ、自前のキーを用意してください。

### インストール手順

1. リポジトリをクローンします：
   ```bash
   git clone https://github.com/bounce12340/pv-signal-monitor.git
   cd pv-signal-monitor
   ```

2. 依存関係をインストールします：
   ```bash
   npm install
   ```

3. 開発サーバーを起動します：
   ```bash
   npm run dev
   ```

4. （任意）アプリ内で自前のAIプロバイダーを設定する場合: 右上の歯車アイコンをクリックし、プロバイダー／モデル／APIキーを入力してください。`.env`ファイルは不要です — キーはブラウザ内にのみ保存され、ビルドに含まれたりアップロードされたりすることはありません。

### テスト

```bash
npm test
```

現在のテストスイート: **13個のテストファイル、94個のテスト**、すべて成功（`vitest run`）。

## 🛠️ 技術スタック
- **フロントエンド:** React 19, Vite, Tailwind CSS v4, Lucide React
- **AI統合:** 単一の統合LLMクライアント（`services/llm.ts`）がリクエストごとに3つのソースのいずれかを解決します——プラットフォームデフォルトの同一オリジンプロキシ、Google GenAI SDK（`@google/genai`、構造化出力）、または汎用のOpenAI互換REST呼び出し（OpenRouter / Ollama / LM Studio）——加えて、AEマスター抽出と文献パイプラインの両方で使われる共通のJSON修復パーサーとバッチ並行処理ヘルパーを1つずつ備えています。
- **文献検索:** PubMed E-utilities（esearch/efetch + XML解析）、LLM層とは独立
- **PV語彙:** シグナル集計とCIOMS/E2Bのフィールドマッピングのための、MedDRA PT → SOCのシード辞書（オフライン参照であり、ライセンスされた完全なMedDRA配布版ではありません）
- **データストレージ:** IndexedDB（単一データベース、単一のキーバリューオブジェクトストア）を採用し、同期読み取り用のインメモリキャッシュ、IndexedDBが利用できない場合のlocalStorageフォールバック、旧localStorageテーブルからの一度限りの移行処理を備えています。
- **バックエンド:** 単一のCloudflare Worker——静的アセット、`/llm/*`プラットフォームLLMプロキシ、`/ollama-cloud/*`互換エイリアス、`/api/*`クロスデバイス同期（D1）、KVによるレート制限
- **テスト:** Vitest

## 🔌 AIプロバイダーに関する注意

設定画面で選択できる3つのソースがあります:

- **プラットフォームデフォルト（キー不要）:** 自前のプロバイダーが設定されていない場合、リクエストは同一オリジンの`/llm/*`へ送られ、Workerが設定済みの上流（OpenAI互換）へプロキシします。ブラウザ側が`Authorization`ヘッダーを送っていない場合に限り、サーバー側のキーが注入されます。設定は不要で、IPごとにレート制限がかかります。
- **自前のOpenAI互換エンドポイント:** ベースURL（＋任意のキー）を指定します。モデル一覧はそのエンドポイントの`GET /models`からリアルタイムに取得され、選択肢として提示されます。テキストと画像の入力に対応しており、PDFはブラウザ内（pdf.js）でテキストレイヤーを抽出して処理します——**純粋なスキャンPDF（画像のみでテキストレイヤーがない場合）は、視覚対応モデルが引き続き必要です**。OpenRouter、ローカルのOllama、プラットフォームのOllama-Cloudエイリアス向けのクイックプリセットが設定ダイアログに組み込まれています。
- **自前のGemini:** ネイティブのPDFアップロードや、スキーマ制約付きの構造化JSON出力（修復処理不要）を含め、フルサポートしています。

## ⚠️ データの永続性

プライマリストアはブラウザの**IndexedDB**（単一データベース、単一オブジェクトストア）で、同期読み取り用のインメモリキャッシュがミラーします。IndexedDBが利用できない場合、アプリはlocalStorageにデグレードします。アップグレード後の初回読み込み時に、既存のlocalStorageデータは一度だけIndexedDBへ移行されます（元のlocalStorageの値はロールバック用のスナップショットとしてそのまま残ります）。

**Library Mode → 匯出備份 (Export Backup)** を定期的に実行し、復元する際はそのJSONファイルをインポートしてください——バックアップにはAEマスターデータに加え、文献ライブラリと未レビューキューも含まれます。Cloudflareにデプロイされている場合（後述）、ナビゲーションバーの同期ウィジェットがスナップショットをD1にも push します（ユーザーごとに直近20件を保持）。これによりクロスデバイス同期とポイントインタイムリカバリが可能になります。

## ☁️ デプロイ（Cloudflare Workers）

アプリは静的アセットを伴う単一のWorkerとしてデプロイされます（`wrangler.jsonc`）：

```bash
npm run build
npx wrangler deploy
```

コンポーネント構成：

- **静的アセット＋SPAフォールバック**は直接配信され、Workerを呼び出すのは `/llm/*`（プラットフォームのLLMプロキシ）、`/ollama-cloud/*`（互換エイリアス）、`/api/*`（同期）のみです。
- **サーバーサイドのLLMキー:** `npx wrangler secret put OLLAMA_API_KEY` により、プラットフォームデフォルトおよびOllama-Cloudエイリアスのルートが、クライアント側のキーなしで上流に対して認証できるようになります。これが安全なのは、このデプロイがカスタムドメインの前段に置かれたアクセス制御レイヤーの内側に完全に位置しているからに限られます — `workers_dev` は `wrangler.jsonc` で意図的に**無効化**されています。`workers.dev` のURLはそのレイヤーを迂回してしまい、Workerを注入済みキーの公開プロキシにしてしまうためです。
- **レート制限:** KVを利用した固定ウィンドウ方式のリミッターが、`/llm/*` と `/ollama-cloud/*` ルートに対してIPごとの1分あたりのリクエスト数を制限します（KVネームスペースが利用できない場合はフェイルオープン＝制限なしで動作）。
- **クロスデバイス同期:** D1データベース（`npx wrangler d1 create pv-signal-monitor` を実行し、`snapshots` テーブルを作成 — `worker/index.ts` を参照）と、ドメイン前段のアクセス制御レイヤーが必要です。ユーザーIDは、そのレイヤーが注入する `Cf-Access-Authenticated-User-Email` ヘッダーから取得されます。
- **CI/CD:** `.github/workflows/ci.yml` がプッシュのたびに型チェック／テスト／ビルドを実行し、`CLOUDFLARE_API_TOKEN`（および任意で `CLOUDFLARE_ACCOUNT_ID`）のリポジトリシークレットが設定されていれば `main` ブランチを自動デプロイします。
