# PV AE 마스터 생성기 및 시그널 모니터 (PV AE Master Generator & Signal Monitor)

의약품 첨부문서에서 이상사례(AE) 데이터를 추출하고, 그 첨부문서를 기준으로 분기별 발생률 시그널 탐지를 수행하며 — 최근 PV-Link 프로젝트와의 병합 이후로는 — 발표된 문헌(PubMed)을 모니터링하여 CIOMS/E2B 초안 작성까지 이어지는 새로운 안전성 시그널을 찾아내는, PV 시그널 탐지 워크플로우 전체를 아우르는 AI 기반 최신 약물감시(Pharmacovigilance, PV) 웹 애플리케이션입니다.

## 🌟 주요 기능

상단 내비게이션은 8개 항목으로 구성되며, **NEW로 표시된 항목**은 PV-Link에서 병합된 문헌 모니터링 워크플로우입니다.

1. **📋 대시보드 (总覽)**
   - 제품별 분기 모니터링 진행 상황을 한눈에 파악하고, 이번 분기에 아직 모니터링을 완료하지 않은 제품으로 바로 이동하는 단축 버튼을 제공합니다.
   - **NEW** — "문헌 검토 대기" 알림(건수 배지)을 표시하며, 클릭하면 문헌 검토 화면으로 바로 이동합니다.

2. **🤖 AI 기반 AE 추출 (생성기 모드)**
   - PDF, 이미지 또는 텍스트 형식의 의약품 첨부문서를 업로드합니다 — 단일 파일 또는 **다중 파일 일괄 대기열**(파일별 재시도가 가능한 순차 추출)로 처리할 수 있습니다.
   - 단일 통합 LLM 클라이언트(`services/llm.ts`)를 통해 처리되는 다중 공급자 AI 지원: **플랫폼 기본값**(키 불필요), **BYO Gemini**(구조화 출력), 또는 임의의 **BYO OpenAI 호환 엔드포인트**(OpenRouter, Ollama, LM Studio 등).
   - 데이터베이스에 저장하기 전에 AI가 추출한 데이터를 검토하고 다듬을 수 있는 편집 가능한 데이터 그리드를 제공합니다(일괄 처리 결과도 저장 전 사람의 검토가 필요합니다).

3. **🔎 문헌 검색 (NEW)**
   - 목표 활성성분, 이상사례(AE) 키워드, 제외어를 조합해 구성하는 정확하고 재현 가능한 **PubMed E-utilities** 검색(esearch + efetch)입니다. 날짜 필터는 발행일이 아니라 PubMed에 **등재된 날짜(entry date, edat)** — 즉 해당 문헌이 PubMed에 수록된 시점 — 을 기준으로 하므로, 모니터링 구간이 새로 나타난 문헌을 안정적으로 포착합니다.
   - 검색된 모든 문헌에 대해 AI 관련성 점수(0~100)와 번체 중국어 요약 + 결론이 생성되며, 통합 LLM 클라이언트를 통해 배치로 동시 처리됩니다.
   - 검색 조건(건수 상한, 날짜 범위, AE 키워드, 제외어)은 세션 간에도 유지되며, 매 라운드마다 목표 성분만 다시 입력하면 됩니다.

4. **✅ 문헌 검토 (NEW)**
   - AI가 관련성 점수를 매기고 요약을 생성한 문헌들이 사람의 판단을 기다리는 대기열이며, 최소 관련성 점수로 필터링할 수 있습니다.
   - 문헌을 선택하면 구조화된 PV 데이터(제품명, 성분, 이상사례 원문(verbatim), MedDRA 기본용어(PT) 후보, 심각성, 인과관계, 결과 등)가 레코드당 한 번씩 자동으로 추출됩니다.
   - **정식 문헌 라이브러리로 입고** 또는 **반려**할 수 있으며, 두 동작 모두 감사 추적에 기록됩니다. 레코드마다 **CIOMS-I / E2B(R3) 초안**을 생성, 조회, 복사 또는 다운로드할 수 있습니다(추출된 데이터로부터의 결정론적 필드 매핑이며, 명시적으로 PV 담당자의 검토가 아직 필요한 초안 골격입니다).

5. **📊 시그널 탐지 및 모니터링 (모니터 모드)**
   - 판매량과 1일 투여량으로 분기별 노출량을 계산하고, 분기별 AE 발생 건수(행마다 **중대(Serious)** 플래그 지정 가능)를 입력합니다.
   - 실제 발생률을 — 정확(Garwood) **Poisson 95% 신뢰구간**과 함께 — 첨부문서 임계값과 비교하여 **예상치 못한 사례(Unexpected)**, **주의(Alert)**, **경고(Warning)** 시그널을 표시합니다.
   - **설정 가능한 시그널 규칙**(최소 사례 수, 경고 배수, 허용 오차 버퍼) — 모든 보고서에는 사용된 규칙이 기록되며, 노이즈로 억제된 행에는 그 사실이 주석으로 표시됩니다. 매칭되지 않은 AE 입력값에 대해서는 퍼지 매칭(및 AI 동의어 정규화 처리)이 가장 가까운 마스터 용어를 제안합니다.
   - CSV로 내보내거나, 규칙 설명과 서명란이 포함된 인쇄용 분기 보고서(PDF로 저장 가능)를 출력할 수 있습니다.
   - **NEW** — 분기 보고서 아래에 독립적인 두 번째 "문헌 시그널 모니터링" 패널이 추가되었습니다. 문헌 라이브러리를 **(활성성분 × MedDRA 기본용어(PT))** 기준으로 집계하며 키워드로 필터링할 수 있습니다. 이는 정성적인 문헌 건수 집계로, 위의 Poisson 신뢰구간 분석과는 다른 방법론이며(그 차이를 명확히 하기 위해 나란히 표시됩니다) — 통계적 유의성 검정이 아닙니다.

6. **📚 문헌 라이브러리 (NEW)**
   - 문헌 라이브러리(검토 모드에서 입고된 레코드)를 조회하고, 키워드/날짜로 필터링하며, CSV로 내보낼 수 있습니다.
   - 구조화된 PV 데이터가 아직 없는 레코드에 대해 진행률 표시기와 함께 **일괄 구조화 추출**을 수행할 수 있습니다.
   - 라이브러리에서 레코드를 제거할 수 있으며(감사 로그에 기록됨), 클릭하면 검토 모드에서 해당 레코드를 다시 열 수 있습니다.

7. **🗄️ 마스터 데이터 관리 (라이브러리 모드)**
   - 저장된 모든 제품 AE 마스터를 관리하는 중앙 데이터베이스입니다.
   - 과거 분기별 모니터링 보고서와 시그널 탐지 배치 기록을 확인할 수 있습니다.
   - **분기 간 추세 분석**: AE별 발생률 추이를 선 그래프로 표시하며, 2개 분기 이상 연속 상승한 용어는 자동으로 플래그됩니다.
   - **마스터 버전 이력**: 업데이트할 때마다 이전 마스터 행이 보관되며, 추가/삭제/임계값 변경 사항을 비교하는 차이 보기를 제공합니다.
   - **전체 JSON 백업 내보내기/가져오기** — 백업 스키마에는 문헌 라이브러리와 검토 대기열도 포함되어 있어, 파일 하나로 전체를 복원할 수 있습니다.

8. **🛡️ 감사 추적 (감사 모드)**
   - **SHA-256 해시 체인**을 적용한 GVP 가이드라인 준수 시스템 로그입니다: 모든 기록이 이전 기록과 연결되며, UI가 체인 무결성을 검증하므로 위변조나 삭제를 탐지할 수 있습니다.
   - 문헌 워크플로우를 포함한 모든 모듈에서 발생하는 `CREATE`, `UPDATE`, `DELETE`, `EXPORT`, `IMPORT`, `ANALYSIS` 동작을 추적하는 읽기 전용 감사 추적입니다.

**⚙️ 설정** (우측 상단 톱니바퀴 아이콘): AI 공급자/모델/키 설정과 시그널 규칙 조정 — 아래 **AI 공급자 안내**를 참고하세요.

## 🚀 시작하기 (로컬 개발)

### 사전 준비
- Node.js (v18 이상 권장)
- 앱을 사용해 보는 데 AI API 키가 필요하지 않습니다 — 플랫폼 기본 프록시는 키가 필요 없습니다. 특정 공급자/모델(Google Gemini, OpenRouter, 로컬 Ollama 등)을 사용하려는 경우에만 직접 키를 준비하면 됩니다.

### 설치 방법

1. 저장소를 클론합니다:
   ```bash
   git clone https://github.com/bounce12340/pv-signal-monitor.git
   cd pv-signal-monitor
   ```

2. 종속성을 설치합니다:
   ```bash
   npm install
   ```

3. 개발 서버를 시작합니다:
   ```bash
   npm run dev
   ```

4. (선택 사항) **앱 내에서** BYO AI 공급자를 설정합니다: 우측 상단의 톱니바퀴 아이콘을 클릭하고 공급자/모델/API 키를 입력하세요. `.env` 파일은 필요하지 않습니다 — 키는 브라우저에만 저장되며, 빌드에 포함되거나 업로드되지 않습니다.

### 테스트

```bash
npm test
```

현재 테스트 스위트: **테스트 파일 13개, 테스트 94개** 모두 통과(`vitest run`).

## 🛠️ 기술 스택
- **프론트엔드:** React 19, Vite, Tailwind CSS v4, Lucide React
- **AI 통합:** 단일 통합 LLM 클라이언트(`services/llm.ts`)가 요청마다 세 가지 소스 중 하나를 선택합니다 — 플랫폼 기본 동일 출처 프록시, Google GenAI SDK(`@google/genai`, 구조화 출력), 또는 범용 OpenAI 호환 REST 호출(OpenRouter / Ollama / LM Studio). 여기에 AE 마스터 추출과 문헌 파이프라인이 함께 사용하는 하나의 JSON 복구 파서와 배치 동시성 헬퍼가 더해집니다.
- **문헌 검색:** PubMed E-utilities(esearch/efetch + XML 파싱)로, LLM 계층과는 독립적으로 동작합니다.
- **PV 어휘체계:** 시그널 집계와 CIOMS/E2B 필드 매핑에 사용되는 MedDRA 기본용어(PT) → 신체기관분류(SOC) 시드 사전(오프라인 조회용이며, 정식 라이선스를 받은 완전한 MedDRA 배포판이 아닙니다).
- **데이터 저장소:** IndexedDB(단일 데이터베이스, 단일 키-값 오브젝트 스토어)를 동기 읽기용 인메모리 캐시와 함께 사용하며, IndexedDB를 사용할 수 없을 때는 localStorage로 대체되고, IndexedDB 도입 이전의 localStorage 테이블에서 한 번만 실행되는 마이그레이션을 수행합니다.
- **백엔드:** 단일 Cloudflare Worker — 정적 자산, `/llm/*` 플랫폼 LLM 프록시, `/ollama-cloud/*` 호환성 별칭, `/api/*` 기기 간 동기화(D1), KV 기반 속도 제한.
- **테스트:** Vitest

## 🔌 AI 공급자 안내

설정에서 선택하는 세 가지 소스:

- **플랫폼 기본값(키 불필요):** BYO 공급자가 설정되어 있지 않으면 요청은 동일 출처의 `/llm/*`로 전송되며, Worker가 설정된 업스트림(OpenAI 호환)으로 프록시하고 — 브라우저가 자체 `Authorization` 헤더를 보내지 않은 경우에 한해 — 서버 측 키를 주입합니다. 별도 설정이 필요 없으며, IP 기준으로 속도 제한이 적용됩니다.
- **BYO OpenAI 호환:** 기본 URL(및 선택적 키)을 입력하면, 해당 엔드포인트의 `GET /models`에서 실시간으로 모델 목록을 가져와 선택지로 제공합니다. 텍스트와 이미지 입력을 지원하며, PDF는 브라우저 내에서 텍스트 레이어를 추출하는 방식(pdf.js)으로 처리됩니다 — **텍스트 레이어가 없는 순수 스캔 PDF(이미지만 있는 경우)는 여전히 비전 지원 모델이 필요합니다**. 설정 대화상자에는 OpenRouter, 로컬 Ollama, 플랫폼의 Ollama-Cloud 별칭에 대한 빠른 사전 설정이 내장되어 있습니다.
- **BYO Gemini:** 네이티브 PDF 업로드와 스키마 제약 구조화 JSON 출력(별도 복구 단계 불필요)을 포함해 완전히 지원됩니다.

## ⚠️ 데이터 내구성

주 저장소는 브라우저의 **IndexedDB**(단일 데이터베이스, 단일 오브젝트 스토어)이며, 동기 읽기를 위한 인메모리 캐시가 이를 미러링합니다. IndexedDB를 사용할 수 없으면 앱은 localStorage로 저하됩니다. 업그레이드 후 처음 로드할 때 기존 localStorage 데이터가 IndexedDB로 한 번만 마이그레이션됩니다(원본 localStorage 값은 롤백 스냅샷으로 그대로 남습니다).

**라이브러리 모드 → 匯出備份 (Export Backup)** 기능을 정기적으로 사용하고, 복원이 필요할 때는 해당 JSON 파일을 가져오기 하세요 — 백업에는 AE 마스터 데이터와 함께 문헌 라이브러리와 검토 대기열도 포함됩니다. Cloudflare에 배포된 경우(아래 참조) 내비게이션 바의 동기화 위젯이 추가로 D1에 스냅샷을 전송하며(사용자별 최근 20개 보관), 기기 간 동기화와 특정 시점 복구가 가능해집니다.

## ☁️ 배포 (Cloudflare Workers)

이 앱은 정적 자산과 함께 단일 Worker로 배포됩니다(`wrangler.jsonc`):

```bash
npm run build
npx wrangler deploy
```

구성 요소:

- **정적 자산 + SPA 폴백**이 직접 제공되며, `/llm/*`(플랫폼 LLM 프록시), `/ollama-cloud/*`(호환성 별칭), `/api/*`(동기화)만 Worker를 호출합니다.
- **서버 측 LLM 키:** `npx wrangler secret put OLLAMA_API_KEY`를 실행하면 플랫폼 기본값 및 Ollama-Cloud 별칭 라우트가 클라이언트 측 키 없이도 업스트림에 인증할 수 있습니다. 이는 배포가 커스텀 도메인 앞단의 접근 제어 계층 뒤에 완전히 위치하기 때문에만 안전합니다 — `wrangler.jsonc`에서 `workers_dev`를 의도적으로 **비활성화**해 두었는데, `workers.dev` URL은 그 접근 제어 계층을 우회하여 Worker를 주입된 키에 대한 오픈 프록시로 만들어 버리기 때문입니다.
- **속도 제한:** KV 기반 고정 윈도우 리미터가 `/llm/*`와 `/ollama-cloud/*` 라우트에서 분당 IP별 요청 수를 제한합니다(KV 네임스페이스를 사용할 수 없으면 열린 상태로 실패하여, 즉 제한 없이 통과시킵니다).
- **기기 간 동기화:** D1 데이터베이스가 필요하며(`npx wrangler d1 create pv-signal-monitor` 실행 후 `snapshots` 테이블 생성 — `worker/index.ts` 참고), 도메인 앞단의 접근 제어 계층도 필요합니다. 사용자 식별은 해당 계층이 주입하는 `Cf-Access-Authenticated-User-Email` 헤더로 이루어집니다.
- **CI/CD:** `.github/workflows/ci.yml`이 모든 push마다 타입 체크/테스트/빌드를 실행하며, `CLOUDFLARE_API_TOKEN`(및 선택적으로 `CLOUDFLARE_ACCOUNT_ID`) 저장소 시크릿이 설정되어 있으면 `main`을 자동으로 배포합니다.
